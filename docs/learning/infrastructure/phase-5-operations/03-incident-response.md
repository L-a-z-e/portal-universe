# 인시던트 대응 프로세스

체계적인 인시던트 대응 프로세스를 수립합니다.

---

## 학습 목표

- [ ] 인시던트 심각도를 분류할 수 있다
- [ ] 대응 프로세스를 이해하고 실행할 수 있다
- [ ] 역할을 분담하고 협업할 수 있다

---

## 1. 인시던트 심각도

### 분류 기준

| 레벨 | 이름 | 기준 | 대응 시간 |
|------|------|------|----------|
| **P1** | Critical | 전체 서비스 중단 | 즉시 |
| **P2** | High | 핵심 기능 장애 | 15분 내 |
| **P3** | Medium | 일부 기능 장애 | 1시간 내 |
| **P4** | Low | 사소한 문제 | 업무 시간 |

### Portal Universe 예시

| 상황 | 심각도 |
|------|--------|
| API Gateway 다운 | P1 |
| 인증 서비스 장애 | P1 |
| 검색 기능 느림 | P2 |
| 알림 지연 | P3 |
| 관리자 대시보드 오류 | P4 |

## 2. 대응 프로세스

### 단계별 흐름

```
┌──────────────┐
│   1. 감지    │ ← 알림, 모니터링, 사용자 신고
└──────┬───────┘
       ↓
┌──────────────┐
│   2. 평가    │ ← 심각도 결정, 영향 범위 파악
└──────┬───────┘
       ↓
┌──────────────┐
│   3. 소집    │ ← 필요 인원 호출, 역할 배정
└──────┬───────┘
       ↓
┌──────────────┐
│   4. 대응    │ ← Runbook 실행, 복구 작업
└──────┬───────┘
       ↓
┌──────────────┐
│   5. 해결    │ ← 정상화 확인, 알림 해제
└──────┬───────┘
       ↓
┌──────────────┐
│   6. 후속    │ ← 인시던트 기록, Post-mortem
└──────────────┘
```

---

## 3. 역할 분담

### 주요 역할

| 역할 | 책임 |
|------|------|
| **Incident Commander (IC)** | 전체 조율, 의사결정 |
| **Operations Lead** | 기술 대응 지휘 |
| **Communications Lead** | 상황 공유, 고객 안내 |
| **Scribe** | 타임라인 기록 |

### 소규모 팀

- IC + Operations: 1명 (겸임)
- Scribe: 1명
- 실제 대응: 1-2명

---

## 4. 커뮤니케이션

### 내부 채널

```
#incident-active: 실시간 대응 논의
#incident-updates: 상태 업데이트 (5-10분 주기)
```

### 상태 업데이트 템플릿

```markdown
## 상태 업데이트 [HH:MM]

**현재 상태**: 조사 중 / 완화 중 / 모니터링 중 / 해결됨

**영향**: [영향받는 기능/사용자]

**진행 상황**:
- [수행한 작업]
- [다음 단계]

**예상 복구 시간**: [시간 또는 "확인 중"]
```

## 5. 인시던트 타임라인 예시

```
14:00 - 알림 수신: API Gateway ServiceDown
```
